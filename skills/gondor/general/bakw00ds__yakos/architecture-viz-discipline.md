# Architecture viz discipline

Loads when Claude reads `.yakos.yml` (to check
`profile.standards.architecture-viz`) or any architecture
artifact file.

## What this rule is for

If `profile.standards.architecture-viz == true`, the project
maintains a navigable architecture surface in the UI that
visualizes:

- Overall system architecture (C4 system + container diagrams)
- Per-component conceptual diagrams
- ADRs (architecture decision records)
- Project phases / status (inferred from CHANGELOG version
  milestones)
- Technical debt log
- Novel engineering / design / AI capability highlights
- Software bill of materials (SBOM)

The architecture page is a READING of these existing artifacts
into a single navigable UI surface. The skill scaffolds the
renderer; the operator (and `architect` agent) maintain the
source files.

## What's required

- **`docs/architecture/` directory** with at least:
  - `system.mmd` — C4 System Context (L1) Mermaid source
  - `container.mmd` — C4 Container (L2)
  - `component.mmd` — C4 Component (L3) per major component
- **`docs/adr/` directory** with at least one ADR (numbered:
  `0001-<slug>.md`, `0002-...`). yakOS `skill:adr-write`
  enforces ADR shape.
- **`docs/tech-debt.md`** — operator-maintained debt log.
  Format: H2 entries with severity / area / cost-to-fix /
  references.
- **`docs/novel-capabilities.md`** — operator-maintained
  "what's special here" log. Format: H2 entries with what /
  why-novel / references.
- **A rendered architecture page** at
  `<project>/docs/architecture.html` (static) or routed via
  the project's frontend framework. Generated by
  `skill:architecture-viz-scaffold`.

## What's updated when

Per `rule:lead-dispatch-discipline`, the lead dispatches the
`architect` agent (already shipped in yakOS at
`lib/agents/architect.md`) for material architecture work.
The architect's discipline:

- **Before a material architecture change merges**: update or
  add the relevant ADR; update affected `.mmd` diagrams; add a
  tech-debt entry if the change defers debt rather than paying it.
- **On novel-capability addition**: add an entry to
  `novel-capabilities.md`.
- **Re-generate the architecture page** (run
  `skill:architecture-viz-scaffold` regenerate-step) so the UI
  reflects current state.

## What's forbidden

- **Stale diagrams.** A `.mmd` file older than the oldest
  modified service directory is stale by definition. Audit-time
  check fires P2.
- **PNG-only diagrams.** Source-of-truth must be Mermaid
  (`.mmd`) or other text-based DSL. PNG outputs are
  build-artifacts, not source.
- **ADRs without status.** Each ADR has a `Status:` line
  (`Proposed` / `Accepted` / `Superseded by ADR-N` / `Deprecated`).
- **Architecture decisions in PR descriptions only.** PRs are
  ephemeral; the ADR is the durable record.

## Composition with other standards

- **Standard 2 (changelog-ui)**: architecture page links to
  `/changelog` for version-driven phase tracking
- **Standard 6 (about-page)**: about page may collapse into
  the architecture page as a hero/intro section
- **SBOM**: composes with already-shipped
  `skill:sbom-generate`. Architecture page reads
  `sbom.spdx.json` if present

## Anti-pattern

The single most common failure mode: the architecture-viz
scaffold runs once, the page renders beautifully, and then the
diagrams fossilize while the code evolves. The audit-time check
(`lib/playbooks/04-docs-architecture.md` §Architecture viz
freshness) compares diagram mtimes against service-directory
mtimes. Diagrams older than the youngest modified service =
P2 finding.
